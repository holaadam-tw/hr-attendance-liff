# 變更提案：外勤里程表起訖登錄（fieldwork-odometer）

## 背景

大正科技業務（如 E826 吳岳樺）的外勤補貼以「實際行駛公里數」計算。現有外勤打卡（fieldwork.html）僅在離開打卡時讓員工**自填一個里程數字**（`field_work_logs.mileage`），沒有任何佐證與查核依據：

- 沒有「出發前登錄里程表讀數」的起點
- 沒有里程表照片佐證
- 自填數字無法與任何客觀資料比對，管理端只能照單全收

## 目標

1. **出發登錄**：業務出發前先登錄里程表讀數＋拍里程表照片（含 GPS、時間戳）
2. **到站登錄**：每站到達客戶時填當下里程表讀數＋拍里程表照片（現有到達打卡流程內完成）
3. **自動計算區間公里**：系統以「本站讀數 − 前一個讀數點」自動算出每站區間公里數，取代人工自填
4. **GPS 交叉比對**：每個區間同時計算 GPS 打卡點之間的直線距離（Haversine），區間公里明顯異常時（低於直線距離、或高出直線距離數倍）在管理端標示 ⚠️ 供查核
5. **收工登錄（選配步驟）**：當日結束時登錄結束讀數＋拍照，自動算出當日總公里（含回程），供補貼計算
6. **管理端查核**：外勤審核明細顯示出發/到站讀數、區間公里、GPS 直線距離與警示；CSV 匯出加入相關欄位

## 範圍

### 本次包含
- 新資料表 `field_work_trips`（每日行程：出發/收工里程表登錄）
- `field_work_logs` 加欄位：`trip_id`、`odometer_reading`、`odometer_photo_url`、`segment_km`、`gps_distance_km`
- fieldwork.html：行程卡片（出發登錄／收工登錄）、到達打卡整合里程表讀數與照片、今日摘要改用自動計算的公里數
- modules/settings.js：外勤審核列表/明細/CSV 匯出加入里程表資料與 GPS 比對警示

### 本次不包含（後續另案）
- **補貼金額計算**（× 元/km 費率、併入薪資）：本次先把「可查核的公里數資料」記錄完整，費率與薪資整合待業主確認後在 payroll 模組另案
- 越南文介面（業務職位目前皆中文使用者）
- field_work_trips / field_work_logs 的 RLS 收緊（隨 RLS Phase 2/3 一併處理，維持與現有 field_work_logs 相同的存取模式）

## 相容性

- **無行程也能打卡**：未做出發登錄時，到達打卡維持現行流程（自填 mileage），不擋既有使用者
- 既有 `mileage` 欄位保留：舊資料照常顯示；新流程下 `segment_km` 優先顯示
- 功能入口不變：仍走 index.html「外勤/業務」格子（職位=業務 gate 已存在）

## 風險

| 風險 | 對策 |
|------|------|
| 員工亂填讀數 | 讀數必須遞增（≥ 前一讀數點）；單一區間 > 200km 需二次確認；里程表照片佐證；GPS 直線距離交叉比對警示 |
| 忘記出發登錄 | 到達打卡時若無當日行程，提示「尚未出發登錄，本站將無法自動計算區間公里」但不阻擋 |
| 照片上傳失敗導致流程卡死 | 照片為佐證非必填欄位，上傳失敗僅提示，讀數仍可送出 |
| 多租戶 | `field_work_trips` 帶 `company_id`；員工端一律 `.eq('employee_id', currentEmployee.id)`；管理端 `.eq('company_id', window.currentCompanyId)` |
