# 變更提案：外勤行程地圖＋追蹤模式（fieldwork-tracking）

## 背景

fieldwork-odometer（已上線）留下每日行程的離散 GPS 點：出發點、各站到達/離開打卡點、收工點。業主希望進一步「抓取業務整天的行程」。

**技術前提（已與業主確認）**：LINE 不提供任何 API 讓第三方持續讀取用戶位置；LIFF 網頁只有在**開啟且前景**時能取得 GPS。因此本案做的是 LINE 生態內的上限——頁面開著時連續記軌跡、關閉時只剩打卡點——**不是**全天不間斷追蹤（那需要原生 App 或車機）。

## 目標

1. **員工端追蹤模式**：有進行中行程（open trip）時，外勤頁前景期間每 60 秒自動記錄一個 GPS 軌跡點，批次上傳到新表 `field_work_trackpoints`；切背景/鎖屏自動暫停，回前景自動續傳；收工即停
2. **管理端行程地圖**：外勤審核加「🗺️ 行程地圖」，用 Leaflet + OpenStreetMap（零 API 費用、無金鑰）畫當日路線：出發點 → 軌跡路段（實線）→ 各站打卡點（標時間/客戶/區間公里）→ 收工點；沒有軌跡的路段用虛線直連表示「該段僅有起訖點」
3. **資料治理**：軌跡點量大（前景 8 小時 ≈ 480 點/人/日），保留 90 天自動清理（pg_cron）

## 範圍

### 本次包含
- 新表 `field_work_trackpoints`（migration 094，含 company_id、90 天清理排程）
- fieldwork.html：追蹤模式（計時器＋visibilitychange＋批次上傳＋「📡 追蹤中」狀態顯示與告知文案）
- admin.html + modules/settings.js：行程地圖 modal（Leaflet lazy load，只在開地圖時載入）

### 本次不包含
- 原生 App / 背景定位（LINE 內做不到）
- 軌跡點的 RLS 收緊（隨 Phase 2/3，維持與 field_work_trips 相同模式）
- 地圖路線最佳化 / 道路吸附（直接連點，不呼叫 routing API）

## 員工知情

追蹤模式首次啟動時於行程卡片顯示固定說明：「出發後，此頁面開啟期間會自動記錄位置軌跡，供公司核算外勤補貼」。記錄範圍僅限工作行程（出發登錄～收工），符合比例原則。

## 風險

| 風險 | 對策 |
|------|------|
| 軌跡斷斷續續造成誤解 | 地圖上無軌跡路段以虛線明示；管理端說明文字「虛線＝該段僅有起訖打卡點」 |
| 軌跡點暴量 | 60 秒間隔＋精度 >1000m 丟棄＋90 天自動清理＋批次上傳（每 3 點或 3 分鐘一次） |
| 耗電疑慮 | 僅前景記錄（本來就開著頁面才會記），取位置用低精度模式即可滿足畫線需求 |
| CDN 依賴（Leaflet） | 只在管理端開地圖時 lazy load；載入失敗顯示文字版行程清單 fallback |
| 多租戶 | trackpoints 帶 company_id；員工自查 employee_id；管理端 company_id filter |
